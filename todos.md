For us with a ongoing project, we can use `cdk flags --set --unconfigured --recommended` to set the recommended feature flags. This will ensure that our stacks are using the latest features and best practices recommended by AWS CDK. We should also periodically check for new recommended feature flags and update our stacks accordingly to take advantage of new features and improvements.

This check is not meant to generate files and update the feature branch, but rather to catch the fact that a developer made changes that needs a rebuild, but they forgot to do so. This will help ensure that our generated files are always up-to-date and consistent with our codebase. One could use a husky commit hook to automatically build the generated files. But the pull request check might be sufficient. We can also use a `.gitattributes` file to tell GitHub to treat certain files as binary (<https://docs.github.com/en/repositories/working-with-files/managing-files/customizing-how-changed-files-appear-on-github>), which will prevent them from being rendered in the diff view and make it easier to see changes in our codebase. This will help us maintain a clean and organized codebase, and make it easier for developers to understand the changes being made. We can also exclude the generated files from GitHub's language statistics to avoid skewing the statistics and give a more accurate representation of our codebase.
